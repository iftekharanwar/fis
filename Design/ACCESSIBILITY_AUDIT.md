# Accessibility Audit — June 2026

Six-dimension audit (screen reader, dynamic type, motion, hearing, motor, vision)
of the full app, every blocker/major finding adversarially verified against the code.
Color/halation work (High Legibility mode) predates this audit and is done.

Severity rubric — **blocker**: a user group cannot complete the core loop
(open app → pick sport → play a scenario → read result); **major**: severely
degraded; **minor**: polish.

## Status (Phases 1–4 + minors pass — all shipped on branch `Accessibility`)

All 21 confirmed blocker/major findings are **resolved**. Of the 18 minors,
the accessibility-relevant ones are resolved too:

- Done in Phases 1–4: verdict-haptic parity, splash/flash/entrance/zoom motion
  gates, lesson close-chip 44pt, splash-retry button trait, lesson/daily art
  hidden, SpriteKit-canvas invert exemption.
- Done in the minors pass: Smart Invert exemption on every photo/illustration
  surface (scenario bg, chapter poster, scenario preview, lesson art, daily art,
  onboarding photo); onboarding body copy 0.72→0.88; stat-strip/HUD VoiceOver
  grouping (VariableStrip, PlayHUDView, SwishView); remaining decorative-motion
  gates (RevealOverlay rise, daily +IQ rise, profile IQ count-up).

Deliberately **not** changed (with reason):
- **Audio scenePhase/interruption lifecycle** — engine-robustness, not
  accessibility; the proper fix (engine restart + loop resume) carries audio
  regression risk best verified on device. Deferred.
- **IntroView ScrollView + VariableStrip fixed height** — both are unreachable
  in the current call-first flow (no call sites). Moot.
- **Crimson MISS metadata line** — deliberate brand semantic; the MISS state is
  already text-led, so no information is color-only.
- **`accessibilityDifferentiateWithoutColor` reads** — the observer hook exists
  (`AccessibilitySettings`); redundant cues already exist everywhere, so there's
  nothing to branch on yet. Hook guards against regression.
- **`brandMark` dead sound asset** — code cleanliness, not accessibility.

## Confirmed findings (verified)

### [BLOCKER] Lesson reader pages only via invisible tap zones — and lessons gate all practice scenarios, blocking the core loop

- **Dimension:** voiceover · **Effort:** medium · **Affects:** VoiceOver users; also Switch Control (no focusable navigation controls at all)
- **Where:** `Sources/Render/LessonReaderOverlay.swift:183-197, 226-248`, `Sources/Render/LessonView.swift:172-186, 219-234`, `Sources/Render/ChapterView.swift:186-201, 107-118`
- **Evidence:** Both lesson surfaces page exclusively through invisible `Color.clear` tap zones: LessonReaderOverlay.swift:183-197 and LessonView.swift:172-186 (`Color.clear .contentShape(Rectangle()) .onTapGesture { back() / advance() }`). `Color.clear` never becomes an accessibility element, so VoiceOver exposes NO next/back control — no buttons, no swipe actions, no adjustable trait, and card changes (`cardIndex += 1` with a `.id(cardIndex)` transition) are never announced nor focused. The one-time coachmark says "TAP TO CONTINUE / Tap left to go back" — instructions only a sighted user can follow. This is core-loop-blocking because ChapterView.swift:186-201 hides every scenario row until the lesson is finished: `} else if lessonRead { ForEach(...) scenarioRow(...) }` with the comment "Practice unlocks only after the lesson is read" — `lessonRead` only flips when the reader reports `finished: true` (ChapterView.swift:112-117), which requires reaching the last card. A blind user cannot reliably reach the last card, therefore never sees a playable scenario for ANY sport. Bonus gap: when the lesson isn't read, the practice section renders literally nothing (no locked-state explanation), so a VoiceOver user exploring ChapterView has no clue scenarios exist.
- **Fix:** Add accessibility paging to both lesson views: `.accessibilityElement(children: .contain)` on the card area plus `.accessibilityAction(named: "Next card") { advance() }` and `.accessibilityAction(named: "Previous card") { back() }` (or visible Previous/Next buttons shown when VoiceOver is running via `accessibilityVoiceOverEnabled` environment). Announce each card on change (`AccessibilityNotification.Announcement("Card 3 of 7. " + headline + body)`) or move `@AccessibilityFocusState` to the headline. In ChapterView, render a non-hidden "Practice — locked. Finish the lesson above to unlock." row instead of nothing when `!lessonRead`.
- **Verifier notes:** Verified in code. LessonReaderOverlay.swift:183-197 pages solely via Color.clear + contentShape + onTapGesture zones; no accessibilityAction/element/adjustable/UIAccessibility/accessibilityFocus anywhere in the file (grep: zero matches), so VoiceOver/Switch Control get no next/back control, and card changes (.id(cardIndex) transition, lines 50/233) are never announced. The only accessible exit is the labeled Close button, which reports finished:false and does not unlock practice; the accessible Begin button renders only on the last card, and every lesson is 6-8 cards (checked all three curricula), so it is unreachable. ChapterView.swift:186-201 hides all scenario rows behind lessonRead (comments confirm intent), which flips only via onClose(finished:true) at ChapterView.swift:112-117. Routing check confirms no bypass: PhysicsGameApp -> RootView -> AppOpenView -> PostSplashRouterView routes all three unlocked sports' scenarios exclusively through the gated ChapterView. Two corrections: (1) LessonView.swift is a diagnostic-only surface (ARCLAB_LAUNCH_TO=lesson, RootView.swift:134-138), not a shipping co-surface — the production surface LessonReaderOverlay has the identical pattern, so the finding stands; (2) VoiceOver's double-tap-and-hold pass-through could theoretically hit the tap zones, but it is undiscoverable (coachmark is visual-only, one-time, hit-testing disabled) and unreliable, matching the finding's "cannot reliably reach" phrasing. Cited line numbers are accurate. Blocker severity is correct: the core loop (pick sport -> play scenario -> read result) is unreachable for VoiceOver/Switch Control users across all sports.

### [BLOCKER] Basketball numpad input dock: fixed 330pt band + frozen 44pt numpad rows — input UI overlaps itself and the court at AX sizes

- **Dimension:** dynamic-type · **Effort:** large · **Affects:** Low-vision users at AX sizes playing the basketball solve scenarios (numpad entry is the only way to shoot)
- **Where:** `Sources/Render/PlayView.swift:389-400`, `Sources/Render/PlayInputView.swift:236-256`, `Sources/Render/PlayInputSingleFieldView.swift:201-220`, `Sources/Render/PlayLayoutMetrics.swift:48-90`
- **Evidence:** PlayView.swift bottomDock: `let height: CGFloat = { if case .outcome = phase { return 480 } else { return 330 } }()` then `dockContent().frame(height: height)` (lines 390-398) — no ScrollView on the surface. Inside that 330pt band, PlayInputView stacks the input cards row (SF Mono 22 value text → ~68pt at AX5), the 4-row numpad, and the SHOOT pill. Each NumpadButton is `.frame(height: 44)` (PlayInputView.swift:249, PlayInputSingleFieldView.swift:214) with a digit label `.font(.sfMono(size: 20))` → ~62pt glyphs at AX5 overflowing their own 44pt rows, so digit hit areas visually overlap adjacent rows while the actual tap zones stay 44pt — taps land on the wrong key relative to what the user sees. The SHOOT button is the same frozen 56pt pill with ~55pt label text. Total intrinsic content at AX5 is roughly double the 330pt band; the VStack overflows upward over the SpriteKit court and HUD. PlayLayoutMetrics.swift hardcodes the same numbers (`bottomReserveIdle: safeBottom + 330`, `bottomReserveOutcome: safeBottom + 480`, `numpadRowHeight`, lines 48-90) with no Dynamic Type input, so even the scene framing math assumes default-size text.
- **Fix:** Make the dock height intrinsic: measure dockContent with a hidden GeometryReader (or use `.fixedSize(horizontal: false, vertical: true)` + `onGeometryChange`) and feed the measured height into propagateLayout instead of the hardcoded 330/480. Give NumpadButton `.frame(minHeight: 44)` and let the row grow; alternatively clamp the numpad digits with `relativeTo: .callout` so keys grow modestly (a numpad digit doesn't need 62pt). Cap the dock at ~60% of screen height and make its content scroll beyond that.
- **Verifier notes:** All cited evidence verified exactly: PlayView.swift:390-398 fixed 330/480 dock band with no ScrollView; NumpadButton .frame(height: 44) at PlayInputView.swift:249 and PlayInputSingleFieldView.swift:214; SHOOT pill frozen at Sizing.pillButtonHeight=56 (Spacing+Tokens.swift:21); PlayLayoutMetrics.swift:48-90 hardcodes safeBottom+330/+480 with no Dynamic Type input; numpad is the only shoot path and isNumpadVisible is never toggled false (no escape hatch). The strongest refutation candidate fails: PhysicsGameApp.swift:27 clamps the app to .dynamicTypeSize(...xxxLarge), but that clamp only affects SwiftUI-environment-resolved fonts (Anton/Barlow via Font.custom relativeTo:). Font.sfMono (Font+Tokens.swift:69-73) bridges a concrete UIFont through UIFontMetrics.scaledFont(for:) with no trait collection, which resolves against the SYSTEM content size category and bypasses the SwiftUI clamp — and every text element in this dock is sfMono. With the 1.10 bodyBoost and 12pt floor in Font+Tokens, glyphs are slightly LARGER than claimed (digit ~68pt at AX5, not ~62; value ~75pt; SHOOT label ~55pt in the 56pt pill). Corrections: total dock content at AX5 is ~488pt vs 330 (~1.5x, not 'roughly double'); the centered .frame(height:330) spills ~79pt BOTH directions — up over the court AND down past the safe-area bottom, pushing the SHOOT pill mostly below the physical screen edge/home-indicator zone (worse than the claimed upward-only overflow); 'taps land on wrong key' holds only at glyph edges (~80pt line boxes on 52pt row pitch overlap neighbors ~14pt each side; glyph centers still align with their 44pt hit zones). Severity blocker stands: at AX4-AX5 the only commit action (SHOOT) is effectively unreachable and there is no alternate input path, so the core loop breaks. Confidence medium only because the UIFontMetrics-bypasses-clamp mechanism and overflow arithmetic are reasoned from documented framework behavior, not observed in a running build at AX sizes.

### [BLOCKER] Lesson cards advance only via invisible tap-zone gestures, and lessons gate all scenarios

- **Dimension:** motor · **Effort:** small · **Affects:** Switch Control and Voice Control users (also VoiceOver); secondary impact on anyone who can't reliably hit unlabeled screen regions
- **Where:** `Sources/Render/LessonView.swift:172-186, 219-234`, `Sources/Render/LessonReaderOverlay.swift:183-197, 228-243`, `Sources/Render/ChapterView.swift:17-25`
- **Evidence:** Both lesson readers advance/go-back exclusively through invisible Color.clear zones with .onTapGesture and no button trait or accessibility action: LessonView.swift tapZones — `Color.clear ... .contentShape(Rectangle()) .onTapGesture { handleBack() }` (left 33%) and `.onTapGesture { handleAdvance() }` (right 67%), lines 172-186; identical pattern in LessonReaderOverlay.swift lines 183-197. The only real Button is the PrimaryButton("Begin") on the LAST card (LessonView.swift:161), which is unreachable without advancing through the cards first. This is core-loop blocking because scenarios are locked until the lesson is read: ChapterView.swift:17-21 — "scenarios are locked until the user consumes the lesson" via `profile.profile.completedLessons.contains(chapter.lesson.id)`. A Switch Control item-scan never lands on the clear zones (no traits, no labels), and Voice Control has nothing to name; the only workaround is point/grid scanning at arbitrary coordinates. Net effect: a switch user cannot finish any lesson, so every chapter's scenarios stay locked and the core loop (pick sport → play a scenario) is impossible.
- **Fix:** Replace the two Color.clear + .onTapGesture zones with Button(action:){ Color.clear.contentShape(Rectangle()) }.buttonStyle(.plain) and add .accessibilityLabel("Previous card") / .accessibilityLabel("Next card") (or keep the gesture for sighted users and add .accessibilityElement + .accessibilityAction(named:) pair on the card container). Apply identically in both LessonView and LessonReaderOverlay. The tap-zone geometry, haptics, and coachmark stay exactly as designed — only the activation becomes AT-visible.
- **Verifier notes:** All cited code verified accurate. LessonView.swift:172-186 and LessonReaderOverlay.swift:183-197 advance/back exclusively via Color.clear + .contentShape(Rectangle()) + .onTapGesture with no accessibility traits, labels, or actions anywhere (grep confirms zero accessibilityAction/AddTraits in either reader). ChapterView.swift:19-21 gating confirmed and is stronger than claimed: practiceList (lines 186-201) renders no scenario rows at all until lessonRead — scenarios are absent, not just locked. No alternate route exists: PostSplashRouterView routes all three live sports through ChapterView; the ungated LevelTypePickerView is reachable only via the ARCLAB_LAUNCH_TO=leveltypes diagnostic env var (commented as a legacy playtest harness). Two minor imprecisions that don't change the verdict: (1) the TopBar Back and overlay ✕ Close are also real buttons, but both abort with finished=false and cannot complete a lesson; (2) "cannot finish any lesson" is strictly slightly overstated — Switch Control point scanning, Voice Control grid, or an accidental synthesized tap on a scanned static-text element (whose activation point falls in the 67% advance zone) could technically advance, but none are discoverable, named, or reliable, and the only hint (the TAP TO CONTINUE coachmark) is visual-only, first-lesson-only, and hit-testing disabled. Given lesson completion is mandatory for every chapter's scenarios in live navigation, the named user groups cannot complete the core loop through any intended or discoverable mechanism. Blocker severity is correct.

### [MAJOR] Shot results, verdicts, and phase changes are never announced — VoiceOver users get silence at every game beat

- **Dimension:** voiceover · **Effort:** medium · **Affects:** VoiceOver users (blind and low-vision)
- **Where:** `Sources/Render/CallPlayView.swift:192-219, 622-647`, `Sources/Render/RevealOverlay.swift:64-74`, `Sources/Render/PlayView.swift:171-202, 384-462`, `Sources/Render/ArcheryCallPlayView.swift:190-221`, `Sources/Render/SoccerCallPlayView.swift:246-266`, `Sources/Render/DailyQuestionView.swift:261-274`, `Sources/Render/CallVerdictView.swift:22-43`
- **Evidence:** grep for `UIAccessibility.post` / `AccessibilityNotification` across Sources/ returns ZERO hits — the app never posts an announcement or moves focus programmatically. Every result reveal is a silent in-place ZStack swap, not a presentation: (1) CallPlayView.swift:216 `phase = .verdict(resolution, wasCorrect: wasCorrect)` swaps the dock to CallVerdictView, then RevealOverlay.swift:64-70 fades in the RIGHT CALL/WRONG CALL card 900ms later with only a haptic (`slideUpHapticCount += 1`); (2) the frozen-at-apex beat (CallPlayView.swift:622-627 `self.phase = .frozen`) silently replaces the dock with the YES/NO buttons — the user who just released has no idea the call prompt appeared; (3) PlayView.swift:187/196 `phase = .outcome(...)` swaps the numpad for SwishView/MissedView — VoiceOver focus is left on numpad keys that no longer exist; (4) ArcheryCallPlayView.swift:219 and SoccerCallPlayView verdict cases do the same; (5) DailyQuestionView.swift:268 `withAnimation { revealed = true }` shows "RIGHT./NOT QUITE." with no announcement. Yes and No buttons below."; on verdict announce the full verdict ("Nailed it. Pair with `@AccessibilityFocusState` to move focus to the verdict headline / YES button when each dock appears.
- **Verifier notes:** Core claim verified: zero UIAccessibility.post / AccessibilityNotification / AccessibilityFocusState hits in Sources/; every cited line number is exact (CallPlayView.swift:216, 622-627; RevealOverlay.swift:64-70; PlayView.swift:187/196; ArcheryCallPlayView.swift:219; SoccerCallPlayView.swift:253-254; DailyQuestionView.swift:268). However, severity is inflated: (1) "silence at every beat" is overstated — distinct outcome SFX (swish/glass/rimDrop vs airball/missTone) fire on every resolution and CallVerdictView:41 fires a .success/.error haptic keyed to call correctness; (2) all verdict content is in the accessibility tree and reachable by scrubbing — MissedView:51 even has an explicit accessibilityLabel for the miss category, and the codebase has deliberate label work elsewhere (HomeView, PlayInputView, ProfileView, CelebrationView), making missing announcements a real gap but not a wall; (3) the frozen call dock (CallPlayView:398-416) has no timeout — no beat is time-limited, so a VoiceOver user can always recover by scrubbing and complete open → pick sport → play → read result. Per the rubric (blocker = cannot complete core loop), this is "severely degraded" (re-scrub required at every beat, phase changes undetectable without exploration), i.e. major, not blocker.
